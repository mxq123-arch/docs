## Summary

Fixes SEO metadata issues found across MDX pages in scope: one short English description and 12 non-English API pages that referenced OpenAPI operations not present in the translated specs.

## Changes

- `api/static-export/overview.mdx`: extended description from 127 → 144 characters so it clears the 130-char minimum.
- Added page-specific `description` frontmatter to 12 translated API reference pages whose `openapi:` operation reference does not resolve to an operation in the corresponding translated spec (so no description is inherited):
 - `{es,fr,zh}/api/analytics/searches.mdx`
 - `{es,fr,zh}/api/analytics/views.mdx`
 - `{es,fr,zh}/api/analytics/visitors.mdx`
 - `{es,fr,zh}/api/analytics/feedback-by-page.mdx`
- All Latin-script descriptions are 130–155 characters; Chinese descriptions are sized to an equivalent visual length. Each description is unique.

## Context

All other in-scope pages either have compliant frontmatter or inherit `title`/`description` from an OpenAPI operation reference whose spec entry defines them, so no further changes were required. No titles were over 60 characters, no descriptions were over 160 characters, and no duplicate descriptions were found among in-scope pages.

---

Note

**Low Risk** 
Documentation-only frontmatter updates; no runtime, API, or auth behavior changes.

**Overview** 
This PR tightens **SEO frontmatter** on 13 MDX pages without changing page body content.

The English **static export** overview gets a longer `description` (adds “for self-hosting”) so it meets the **~130-character minimum** for meta descriptions.

Twelve **localized analytics API** pages (`es`, `fr`, `zh` × searches, views, visitors, feedback-by-page) gain explicit, unique `description` values in frontmatter. Those pages still reference `openapi:` operations that **don’t resolve** in the translated OpenAPI specs, so they weren’t inheriting operation-level metadata before.
